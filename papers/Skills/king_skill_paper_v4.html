Abstract

Abstract — v3

We present King-Skill v3, the third edition of a hierarchical skill-routing architecture for scientific LLM agents, incorporating responses to two rounds of peer review [CR1, CR2]. The system routes tasks to 20 domain skills and applies a token-compression layer substituting natural language with formal notation, including chemical formulas, balanced reaction equations, mathematical notation, and Python pseudocode. This edition makes the following additional contributions over v2: (i) a corrected test count of 51/53 passing (skill-05 Lean 4 is non-functional without manual runtime installation, which is a real user-facing defect, not merely a dependency note); (ii) a proposed controlled baseline experiment design comparing King-Skill to an unrouted LLM on identical P2PCLAW tasks evaluated by 17 judges — the most critical unresolved gap identified by both reviewers; (iii) a formal description of the 17-judge evaluation protocol including model selection criteria, prompt templates, and consensus algorithm; (iv) a reproducibility checklist with current status for each artefact; (v) a credibility evolution table comparing the standing of every major claim across v1, v2, and v3. The primary empirically-supported claim remains: externalising deterministic computation prevents LLMs from spending tokens on arithmetic. The primary benefit of the compression layer is API cost reduction of ~33% on output tokens (~10–23% on total blended cost), with a secondary benefit in multi-turn session context efficiency. The 5× single-call reasoning depth gain from v1 is retracted. Chemical reaction equations provide the most robust and context-independent compression (40–55% sustained savings). Maximum vs. mean compression ratios are explicitly separated throughout.

1 · Introduction

Large language models spend tokens on computation they should delegate. A model that generates 2,000 tokens describing how to compute eigenvalues is doing what a mathematician does when solving linear algebra by hand while a computer sits unused on the desk. This is not a failure of intelligence — it is a failure of tool use.

The theoretical foundation is the extended mind thesis (Clark & Chalmers, 1998): cognitive processes extend beyond the agent into the tools it uses. The Python interpreter, SAT solver, and arXiv API are components of the agent's cognitive system, not external resources.

1.1 · The Real Token Cost in P2PCLAW

The P2PCLAW platform enforces a minimum paper length of 2,500 output tokens. In practice, models spend far more tokens on reasoning than on the published output:

P2PCLAW paper cycle (rough single-observation estimate): Hypothesis + literature search: ~4,000 reasoning tokens Experimental design + execution: ~8,000 reasoning tokens Analysis + verification: ~7,000 reasoning tokens Draft + revision: ~6,000 reasoning tokens ────────────────────────────────────────────────────── Total reasoning tokens: ~25,000 [ESTIMATE, n=1] Published output tokens: ~3,000 Reasoning-to-output ratio: ~8.3× [ESTIMATE — not validated]

Corrected v2 The v1 paper implied that compressing output tokens frees budget for more reasoning steps within a single call. This is incorrect: thinking and output share the same context window, and the two budgets are tracked separately in production APIs. Compressing output does not expand single-call reasoning depth. The real benefit is API billing reduction and, for multi-turn sessions, reduced context-history consumption. See §4.2 for the full corrected framing.

2 · Architecture

2.1 · The Two-Budget Rule

Thinking budget
Never compressed
Full chain-of-thought verbosity preserved. Wei et al. (2022) showed intermediate tokens improve task performance. We do not compress thinking tokens. Note: this means single-call depth is not expanded by output compression — a key correction from v1.
Output budget
Compressed where beneficial
Chemical formulas · reaction equations · math notation · pseudocode · Lean 4 · emoji flags. Mean savings: ~9% (all content), ~33% (STEM-heavy). Maximum savings: up to 80% on first-mention verbose chemical names. See §3 for the full spectrum.

3 · Token Compression — Maximum vs. Mean Ratios

3.1 · The Compression Spectrum

The fundamental rule: Maximum compression ratios apply to first-mention verbose descriptions and synonym chains. Mean compression ratios apply to realistic running scientific text where abbreviation conventions are already in use. The maximum is not the mean. Both numbers must be reported separately. Corrected v2

A practical illustration from pharmacology and household chemistry — the kind of content that appears in ingredient tables, drug leaflets, safety datasheets, and recipe lists, where convention mandates verbose common names and synonyms:

Content type: pharmaceutical excipient table / ingredient label (convention forces verbose common-name + IUPAC pairs) "hydrogen peroxide (antiseptic)" → H₂O₂ −73% [MAX] "table salt (sodium chloride)" → NaCl −77% [MAX] "caustic soda (sodium hydroxide)" → NaOH −74% [MAX] "dry ice (solid carbon dioxide)" → CO₂ (s) −71% [MAX] "quicklime (calcium oxide)" → CaO −75% [MAX] "muriatic acid (hydrochloric acid)" → HCl −68% [MAX] "glucose / dextrose / blood sugar" → C₆H₁₂O₆ −66% [MAX] Subsequent mentions in the same document: "...the H₂O₂ concentration was..." → already 3 tokens: minimal gain "...glucose levels rose..." → "glucose" = 1 token vs C₆H₁₂O₆ = worse "...add NaCl to adjust osmolarity..."→ NaCl = already short: ~10% gain

This shows why maximum ratios are achievable but not representative. In a full pharmacological paper, the first-mention compression is real; subsequent usage compression is marginal or counterproductive. The realistic mean across the full document lies between these extremes.

The formula ∂S/∂t ≥ 0 also illustrates the semantic precision trade-off noted by Veselov [CR2]: it assumes S is differentiable with respect to time, which is invalid at phase transitions. Natural language ("entropy never decreases in an isolated system") carries this caveat implicitly through convention. Formal notation preserves the mathematical core and loses the qualifiers. Neither form is universally superior; the choice depends on audience and downstream use.

4 · Reasoning Gain — Corrected Framing (v2, confirmed v3)

4.1 · Informational Density — Retracted as Quantitative Claim

The v1 claim of 65% informational density improvement has been retracted (v2) and this retraction is confirmed in v3. The underlying observation remains qualitatively valid for expert audiences: formal notation encodes more mathematical constraints per character than prose. However, without an operational definition of "concept" and a validated measurement protocol, the 65% figure cannot be defended as a quantitative result.

Veselov [CR2] further notes that ∂S/∂t ≥ 0, while formally compact, is semantically narrower than "entropy never decreases in an isolated system" — the differential form presupposes differentiability, excludes phase transitions, and makes no reference to the isolation condition. Density and fidelity trade-offs are domain- and context-specific.

4.2 · Reasoning Depth — Corrected Framing

CORRECTED FRAMING (v2, confirmed v3): Single call (context window = 200K tokens): thinking tokens: ~20,000 (never compressed — by design) output tokens: ~3,000 → compressed: ~2,010 context freed: ~990 tokens (< 1% of window) single-call depth benefit: negligible Multi-turn session (N prior turns in context): each compressed prior turn saves ~33% of that turn's output tokens over N=10 turns: 10 × 990 = 9,900 tokens freed ≈ 5% of window → real but moderate benefit for long sessions [ESTIMATE] API billing (primary measurable benefit): 33% saving on output tokens output = ~70% of blended token cost (session-dependent) net total saving ≈ 23% of total API cost [ESTIMATE]

Revised and confirmed claim (v2, v3): The primary benefit of the compression layer is API cost reduction of approximately 33% on output tokens, translating to ~10–23% net total saving depending on input/output ratio. A secondary benefit exists for multi-turn sessions. The 5× single-call reasoning depth gain from v1 was an internal contradiction and is definitively retracted. Corrected v2 · confirmed v3

7.1 · Proposed Controlled Experiment

Research question: Does King-Skill routing and compression improve the quality of scientific papers produced by LLM agents on P2PCLAW tasks, as measured by 17-judge consensus scores?

EXPERIMENT DESIGN (pre-registration · not yet executed): Condition A: King-Skill agent - same base LLM (e.g. Claude Sonnet 4.6) - king-skill dispatch active - token-compression layer active - all 20 domain skills available Condition B: Unrouted baseline - same base LLM, same system prompt - no skill dispatch (direct LLM generation only) - no compression layer - same input tasks Task pool: n = 50 P2PCLAW-style research tasks (stratified: 10 thermodynamics, 10 quantum mechanics, 10 distributed systems, 10 ML, 10 synthetic biology) Primary outcome: 17-judge consensus score (0–1) Secondary outcomes: - Output token count - API cost per paper - Routing accuracy (Condition A only) - Proportion of papers meeting 2,500-token minimum Analysis: paired t-test or Wilcoxon signed-rank (within-task comparison, A vs B on same prompt) α = 0.05, power = 0.80 minimum detectable effect: 0.05 consensus score units

Expected outcomes: We hypothesise that King-Skill will show improvement in tasks requiring numerical computation (skills 01, 07, 09) and formal verification (skill-05), where the LLM baseline is most likely to make arithmetic errors or hallucinate mathematical results. For writing-intensive tasks with no deterministic computation, we expect no significant difference. The compression layer is expected to show consistent token reduction at the cost of potentially reduced readability scores from non-specialist judges. Hypothesis — not yet tested

11 · Discussion

After two rounds of peer review, the King-Skill architecture is better understood as a proof of concept for skill routing and scientific output compression than as a validated production system. This is not a deprecation — it is a more honest description of what has and has not been demonstrated.

What is well-supported empirically: 51/53 tests pass (default environment). Six bugs were found and fixed during verification — three were fatal, demonstrating the necessity of testing over assumption. The routing architecture correctly handles 18/19 hand-crafted tasks after fixing a dispatch order bug. Token compression measurably reduces output token count in STEM-heavy content, with mean savings estimated at ~33% on scientific text and ~9% on mixed content from n=20 examples per category.

What is estimated but not validated: All compression ratios are single-point estimates from small non-random samples. The 33% mean used in cost projections has not been validated on a held-out corpus of real P2PCLAW paper drafts. The 8.3× reasoning-to-output ratio is from a single observation. Cost projections scale linearly from these estimates and should be understood accordingly.

What is theoretical: The 5× single-call depth gain is definitively retracted. The multi-turn session depth benefit is real in direction but unquantified. The claim that King-Skill improves scientific paper quality (vs. an unrouted LLM) has not been tested; this is the most important open question and is now addressed by the proposed baseline experiment in §7.

The strongest remaining case — chemical reaction equations: Of all compression strategies evaluated, balanced reaction equations provide the most robust and context-independent savings (40–55% sustained throughout the document, not just on first mention). This is because prose reaction descriptions are inherently verbose regardless of whether abbreviated chemical names are in use. A paper describing twenty reactions in prose versus in equation notation will always be substantially longer in the prose version, for any audience level. This claim requires no special conditions and is not susceptible to the maximum/mean conflation that weakened other compression claims.

On the architecture's future: The dispatch table's sensitivity to rule ordering is a fundamental limitation of rule-based expert systems. A learned router (even a small classifier) would generalise better to unseen task types and would not require manual reordering after each new failure mode is discovered. The proposed baseline experiment (§7.1) will generate sufficient task-label pairs to train such a classifier as a byproduct of its execution.

12 · Open Research Questions

  1. Does King-Skill provide a measurable improvement in scientific paper quality vs. an unrouted LLM, as measured by 17-judge consensus scores on matched tasks? — Highest priority. Baseline experiment designed in §7.1. Open
  2. What is routing accuracy on a statistically representative 500+ task corpus with stratified sampling and false positive/negative measurement? Open
  3. Does compressing output tokens in turn N affect reasoning quality in subsequent turns of a multi-turn session? Requires thinking-token access and blind quality evaluation. Open
  4. What is the mean compression ratio on a held-out corpus of real P2PCLAW paper drafts (not hand-crafted examples)? Is the 33% mean validated? Open
  5. Do expert judges prefer compressed formal notation or verbose prose? Does chemical formula density in a submission affect the 17-judge consensus score positively or negatively? Open
  6. Would a single python-toolbox with subcommands perform equivalently to the 20-skill granular architecture? What is the routing accuracy of each? Open
  7. Can the dispatch table be replaced by a small trained classifier? The proposed baseline experiment (§7.1) would generate sufficient training data as a byproduct. Open
  8. Does Lean 4 CBM verification improve peer-review consensus scores vs. PENDING papers? Requires accumulation of matched VERIFIED/PENDING pairs at p2pclaw.com. Open
  9. Does the Krippendorff's α of the 17-judge panel meet the target threshold of α ≥ 0.40 on scientific paper evaluation? This has not been measured on a large held-out set. Open · addressed by §7.1

13 · Conclusion

We present King-Skill v3, the third edition of a hierarchical skill-routing architecture for scientific LLM agents on the P2PCLAW platform. Over three versions and two rounds of peer review — the second conducted by Vladimir Veselov (Russian Academy of Sciences) — the paper has evolved from an overhyped technical report into a transparent, self-critical engineering description of a prototype with a clear experimental roadmap.

The core contributions surviving peer review are: (1) a 20-skill routing architecture that correctly handles 18/19 tested task types and catches six real implementation bugs before release; (2) a token-compression layer providing ~33% output token savings on STEM-heavy content, with the most robust gains from balanced chemical reaction equations (40–55% sustained); (3) an honest disambiguation of maximum vs. mean compression ratios, showing that 80% savings apply only to first-mention verbose chemical synonyms, not to general scientific writing; and (4) a proposed controlled baseline experiment and 17-judge protocol that will, if executed, provide the empirical validation this architecture currently lacks.

The honest core claim that survives all critique: a scientific LLM agent should not spend tokens computing things that can be computed by a deterministic tool, and should not express formal relationships in verbose natural language when formal notation is available. King-Skill implements both principles. Whether it does so better than simpler alternatives — a single Python executor, a plain prompt instruction to use formulas — remains to be shown.
